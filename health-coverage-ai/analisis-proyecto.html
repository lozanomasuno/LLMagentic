Resumen Ejecutivo

Health Coverage AI es una aplicación SaaS full-stack que permite a asesores de EPS/seguros consultar en lenguaje natural si un procedimiento médico está cubierto para un afiliado específico. El sistema combina búsqueda semántica vectorial (RAG) sobre documentos de póliza con el modelo GPT-4o de OpenAI para generar respuestas estructuradas, precisas y auditables en tiempo real.

Problema que resuelve
Consulta de cobertura en segundos
Reemplaza la búsqueda manual en documentos de 50+ páginas por una respuesta IA con fuentes citadas
Resultado de análisis
JSON estructurado + trazabilidad
Devuelve: cobertura (cubierto / no cubierto / con condiciones), texto explicativo, documentos fuente y traza del agente
Datos del afiliado
30+ atributos clínicos y financieros
Plan, estado de afiliación, morosidad, carencias, autorizaciones previas, preexistencias
Reglas de negocio en prompt
8 reglas de cobertura
Carencia, exclusiones, estado de pagos, autorización previa, copago, necesidad médica, cita de documentos
Diferenciador clave: El agente no responde desde parámetros del modelo entrenado, sino que recupera fragmentos exactos de los documentos de la aseguradora (RAG) y razona sobre ellos, garantizando que la respuesta siempre esté fundamentada en las cláusulas vigentes de la póliza.

Pipeline de Inteligencia Artificial (RAG + LangGraph)
El núcleo del sistema es un agente RAG (Retrieval-Augmented Generation) implementado como un grafo de estados con LangGraph. El agente no alucina respuestas genéricas: recupera fragmentos exactos de los documentos de cobertura y razona sobre ellos.

Grafo del Agente (LangGraph StateGraph)

START │ ▼ [ Nodo 1: retrieve_context ] │ Entrada: query (str) + affiliate (dict) │ Acción: CoverageRetriever.retrieve(query, n_results=6) │ └─► ChromaDB cosine similarity search │ Salida: chunks: list[RetrievedChunk], steps_trace │ ▼ [ Nodo 2: analyze_coverage ] │ Entrada: chunks + affiliate + query │ Acción: Formatea contexto → build_system_prompt() + build_user_prompt() │ └─► AsyncOpenAI.chat.completions.create(model="gpt-4o", temperature=0.1) │ └─► Parsea JSON response (con fallback regex si el LLM añade markdown) │ Salida: coverage_result, response_text, conditions │ ▼ END → CoverageState final → CoverageAnalysisService → persist → HTTP response

Chunking Semántico de Documentos

ParámetroValorDescripción
Tamaño de chunk80 – 900 caracteresMantiene contexto semántico completo sin superar ventana de embedding
Criterio de splitHeadings Word (H1/H2/H3) + patrones numéricosDivisión por sección documental, no arbitraria por tokens
Metadata por chunkdocument_slug, section, chunk_index, char_countPermite citar el documento y sección exacta en la respuesta
Embedding modelall-MiniLM-L6-v2 (ONNX, local)No requiere API key; rápido para producción moderada
Métrica de distanciaCosine → convertida a similitud [0,1]similarity = max(0, 1 - distance/2)
Top-K recuperados6 fragmentosBalance entre contexto y tamaño de prompt

Ingeniería de Prompts

El prompt de sistema instruye a GPT-4o como especialista senior en coberturas médicas con 8 reglas de negocio explícitas que garantizan respuestas correctas:

  • Regla 1: Si afiliado no está "Activo" o pagos no "Al día" → siempre "no_cubierto"
  • Regla 2: Verificar período de carencia según antigüedad del afiliado
  • Regla 3: Detectar si el servicio requiere autorización previa (DOC2 §4)
  • Regla 4: Verificar que el servicio no esté en lista de exclusiones (DOC2 §5)
  • Regla 5: Aplicar criterios de necesidad médica (DOC3 §2)
  • Regla 6: Determinar copago / coaseguro según plan del afiliado (DOC1 §4)
  • Regla 7: Citar siempre la sección del documento que respalda la conclusión
  • Regla 8: Responder únicamente JSON válido (sin texto extra)

Estructura de Respuesta del LLM

{
  "coverage_result": "cubierto" | "no_cubierto" | "cubierto_con_condiciones",
  "response_text":   "Explicación profesional en español con citas a DOC1 §3.2...",
  "conditions":      ["Requiere autorización previa", "Aplica copago del 20%"]

Frontend — Angular 20 SPA

Estructura de Componentes (Feature-based)

ComponenteResponsabilidadTécnicas usadas
AffiliateSearchComponent Punto de entrada principal. Búsqueda por número de documento con spinner. Signals (signal()), FormsModule, lazy-loaded via Router
AffiliateCardComponent Muestra ficha completa del afiliado: plan, estado, mora, autorizaciones. Input binding (@input()), Material Card
CoverageQueryComponent Formulario de consulta en lenguaje natural (validación min. 10 chars). Signals, HttpClient Observable, Material Form
CoverageResultComponent Muestra resultado (badge coloreado), texto explicativo, fuentes y traza del agente. computed() signals, Expansion panels, Badge coloreado por resultado

Decisiones Técnicas Destacadas

  • Signals sobre NgRx: Estado local de componentes con la nueva API de Signals de Angular 16+, sin boilerplate de Redux.
  • Zoneless change detection: provideZonelessChangeDetection() — elimina Zone.js para mejor rendimiento.
  • Lazy loading: AffiliateSearchComponent (y sus hijos transitivos) se cargan solo al navegar a la ruta.
  • Error interceptor HTTP: Centraliza manejo de errores 0 (red), 404, 501, 503 y 5xx con mensajes en español.
  • Proxy nginx: En producción, las llamadas /api/ se enrutan al backend por nginx, evitando CORS y centralizando el tráfico.
